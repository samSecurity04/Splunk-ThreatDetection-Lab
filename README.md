# Splunk Threat Detection Lab

A hands-on threat detection lab built in **Splunk Cloud**, covering three detection projects: impossible-travel (geo-velocity) detection, threat-intel IOC correlation, and behavioral DNS-tunneling detection. Each project is framed as an incident story — scenario, hunt, finding, response, MITRE mapping, and an honest account of limitations and evasion.

> **Read this first — honesty statement.**
> All data in this lab is **synthetic**, generated by Python scripts included in this repo. Detection thresholds were **hand-tuned, not baselined** against production traffic. Where a detection reports "0 false positives," that is **by construction of the synthetic data**, not a validated real-world result. The most valuable part of this lab is **Project 3**, where the DNS detection was tested against a **real labeled dataset** and largely failed — that failure, and the diagnosis of *why*, is documented in full rather than hidden. The war-stories and failures throughout are kept deliberately visible; they are the most credible content here.

## Project 1 — Impossible Travel Detection

**Scenario.** A user account authenticates from two cities so far apart, so close together in time, that the implied travel speed is physically impossible — the classic signal of credential theft or session hijacking, where an attacker logs in from their own location while the legitimate user is active elsewhere.

**The data.** A synthetic auth log (`auth_logs.csv`) for five users across five cities (Melbourne, London, New York, Mumbai, Toronto), each with known devices and normal login patterns. Planted anomalies: Sam logging in Melbourne → London, and Vineet bouncing New York ↔ Mumbai, within impossibly short windows.

📸 [screenshot: 01_travel_sequence.png — impossible_travel_data_generator.py, showing the city coordinate table, known-devices map, and event schema]

📸 [screenshot: 02_travel_spl.png — auth_logs.csv uploaded to Splunk Cloud via Add Data]

📸 [screenshot: 03_travel_detection.png — Set Source Type preview confirming lat/lon/city/device fields parsed correctly]

**The hunt.** The detection computes geo-velocity **live in SPL** — it does not read a pre-labeled field. For each user, `streamstats` pulls the previous login's city, coordinates, and timestamp; the Haversine formula computes great-circle distance; distance ÷ time gives speed in km/h; anything faster than **900 km/h** (faster than a commercial jet) is flagged and assigned a severity.

```spl
index=main sourcetype=csv
| sort 0 user _time
| streamstats current=f last(city) as prev_city last(lat) as prev_lat last(lon) as prev_lon last(_time) as prev_time by user
| where city!=prev_city
| eval time_diff_hours=(strptime(_time,"%Y-%m-%dT%H:%M:%S")-strptime(prev_time,"%Y-%m-%dT%H:%M:%S"))/3600
| eval lat1_rad=prev_lat*3.14159/180, lat2_rad=lat*3.14159/180
| eval dlat=lat2_rad-lat1_rad, dlon=(lon-prev_lon)*3.14159/180
| eval a=sin(dlat/2)*sin(dlat/2)+cos(lat1_rad)*cos(lat2_rad)*sin(dlon/2)*sin(dlon/2)
| eval distance_km=round(2*6371*asin(sqrt(a)),0)
| eval speed_kmh=round(distance_km/time_diff_hours,0)
| eval severity=case(speed_kmh>5000,"Critical",speed_kmh>2000,"High",speed_kmh>900,"Medium")
| where speed_kmh>900
| table _time user prev_city city distance_km time_diff_hours speed_kmh severity
```

**The finding.** Seven events flagged. The standouts:

| User | Route | Distance | Time | Speed | Severity |
|---|---|---|---|---|---|
| Sam | Melbourne → London | 16,903 km | 0.25 h | **67,612 km/h** | Critical |
| Vineet | New York → Mumbai | 12,538 km | 0.38 h | **32,708 km/h** | Critical |
| Vineet | Mumbai → New York | 12,538 km | 0.33 h | **37,614 km/h** | Critical |

Sam at 67,612 km/h is ~55× the speed of sound — unambiguously a hijacked session, not a traveler.

📸 [screenshot: 04_travel_dashboard.png — the finished four-panel dark dashboard]

**Response.** In a real SOC this fires an alert into an analyst queue: force re-authentication, invalidate active sessions, check whether MFA was satisfied on the anomalous login, and confirm with the user out-of-band. The dashboard's investigation-queue panel models this triage view.

**Dashboard.** A four-panel view visualizing this detection's dataset: a global auth-activity map, a login-activity timeline, an event-category severity breakdown, and an analyst investigation queue.

> **Scope note (honest).** The impossible-travel **speed calculation is computed live** in SPL, as shown above. The dashboard also surfaces other auth-anomaly categories — *failed-login burst*, *off-hours new-device*, *impossible-travel MFA failure* — via `stats count by event_type`. Those categories are **labels seeded in the synthetic generator**, counted for visualization; they are **not** independently computed detections. Only the geo-velocity logic is a real detection.

**MITRE ATT&CK.** T1078 (Valid Accounts), T1550 (Use of Alternate Authentication Material).

**Limitations & evasion.**
- **VPNs and proxies are the dominant real-world confounder.** A legitimate user on a VPN can appear to teleport at 60,000 km/h; an attacker proxying through the victim's own city looks entirely local and evades the detection completely. In production this rule is noisy without VPN/egress-IP allowlisting and is trivially bypassed by a geographically-aware attacker.
- Thresholds (900 km/h, severity bands) are hand-picked, not baselined against real login telemetry.
- The "impossible" event types beyond geo-velocity are seeded, not detected (see scope note).
- Detection is pairwise on consecutive logins; a slow "possible but suspicious" drift wouldn't trip it.

## Project 2 — Threat Intel IOC Correlation

**Scenario.** An internal host is quietly beaconing to command-and-control infrastructure. The goal: catch it by correlating outbound network traffic against a **live, real-world** threat-intel blocklist.

**The data.**
- A **real** IOC feed: the [abuse.ch Feodo Tracker](https://feodotracker.abuse.ch/) C2 IP blocklist, cleaned of comment lines and loaded into Splunk as a lookup table (`threat_intel_feed.csv`). Fields: `first_seen_utc, dst_ip, dst_port, c2_status, last_online, malware`.
- Synthetic network traffic (`network_traffic.csv`) for five internal hosts, with planted beacons from one compromised machine.

📸 [screenshot: project2-01-feed-prep-grep.png — grep stripping comment lines from the abuse.ch blocklist into threat_intel_feed.csv, then head -5 confirming the C2 IPs (Emotet, QakBot)]

📸 [screenshot: project2-02-generator-script.png — threat_intel_traffic_generator.py, showing benign traffic plus the planted beacons from Andrea's host]

📸 [screenshot: project2-03-generator-run.png — generator run: 688 network events, threat connections planted to known-bad IPs, compromised user Andrea (10.0.1.52)]

📸 [screenshot: project2-04-upload-traffic.png — network_traffic.csv uploaded to Splunk]

📸 [screenshot: project2-05-raw-events.png — 688 raw traffic events indexed]

📸 [screenshot: project2-06-upload-threat-feed-lookup.png — threat_intel_feed.csv defined as a Splunk lookup table]

**The hunt.** Correlate every outbound destination IP against the C2 lookup. Any traffic whose `dst_ip` matches a known-bad IP is enriched with the malware family and C2 status, then aggregated by host.

```spl
index=main source="network_traffic.csv"
| lookup threat_intel dst_ip OUTPUT malware as threat_name c2_status last_online
| where isnotnull(threat_name)
| stats count as connection_count, sum(bytes_out) as total_bytes by user src_ip dst_ip threat_name c2_status
| sort -connection_count
```

**The finding.** One host — **Andrea, 10.0.1.52** — was beaconing to **four** known C2 servers:

| dst_ip | Malware | C2 Status | Connections |
|---|---|---|---|
| 162.243.103.246 | Emotet | offline | 8 |
| 178.62.3.223 | QakBot | offline | 8 |
| 34.204.119.63 | QakBot | offline | 8 |
| 50.16.16.211 | QakBot | **ONLINE** | 5 |

One C2 was still live at detection time — an active, ongoing compromise, not just historical residue.

📸 [screenshot: project2-07-correlation-4-c2-hits.png — the correlation result: Andrea's host matched against 4 C2 IPs, one online]

📸 [screenshot: project2-08-exfil-by-malware.png — bytes exfiltrated grouped by malware family]

**War-story.** The lookup initially returned **"no results"** — not because the correlation was wrong, but because the query was run before the lookup table had finished saving/propagating in Splunk Cloud. Re-running after the save completed returned the matches. A reminder that in Splunk Cloud, lookup availability is not instantaneous.

📸 [screenshot: project2-09-job-expired-warstory.png — search job expired/canceled during a long timechart run]

**Response.** Isolate 10.0.1.52 from the network, capture volatile memory, hunt for the Emotet/QakBot persistence mechanisms, and block the live C2 IP at the egress firewall while the host is rebuilt.

**MITRE ATT&CK.** T1071 (Application Layer Protocol / C2), T1041 (Exfiltration Over C2 Channel), TA0011 (Command and Control).

**Limitations & evasion (important — read this).**
- **This is IOC/IP-match detection only — it is not C2 detection.** It catches traffic to *known-bad IPs that already appear in a public feed*. It does **not** detect C2 behaviorally.
- Any attacker using **fresh infrastructure** not yet in abuse.ch, **domain fronting**, **fast-flux DNS**, or a **legitimate cloud service as a redirector** evades this completely.
- The feed is a point-in-time snapshot; C2 IPs rotate constantly, so both false negatives (new infra) and stale entries are expected.
- Detection is only as good as the feed's freshness and coverage.

## Project 3 — DNS Tunneling Detection (the star)

> This is the most important project in the lab — not because it worked, but because it was tested against **real data and largely failed**, and the failure was diagnosed honestly instead of buried.

**Scenario.** DNS tunneling smuggles data or C2 traffic inside DNS queries — encoding payload into long, high-entropy subdomains under an attacker-controlled domain. It slips past controls that trust DNS. The goal: detect it **behaviorally**, by computing per-query features live in SPL, not by reading pre-labeled data.

**The data.** Synthetic DNS logs (`dns_logs.csv`): 1,338 queries, of which 120 are a planted tunneling burst from Andrea (10.0.1.52). Normal subdomains are short and low-diversity; tunneling subdomains are long and character-diverse.

📸 [screenshot: project3-01-generator-run.png — dns_exfiltration_data_generator.py run: 1,338 events, 120-query tunneling burst planted from Andrea]

📸 [screenshot: project3-02-event-type-counts.png — stats count by event_type: 120 dns_tunneling vs 1,218 normal_dns]

**The features.** Two computed live from the raw query string:
- **subdomain length** — tunneling packs data into long labels.
- **character diversity** — `unique_chars / subdomain_length`, a 0–1 ratio; encoded payload uses a wider character set than human-readable domains.

**War-story #1 — the aggregate-then-filter trap.** The first attempt aggregated to *per-user averages* and then filtered — `avg(entropy) > 3.5 OR avg(length) > 20 by user`. It returned **zero results**: 120 malicious queries averaged against 1,218 normal ones never cleared the threshold at the user level. The fix: filter at the **per-query** level *first*, then aggregate. Filter-then-aggregate, not aggregate-then-filter.

📸 [screenshot: project3-03-noresults-avg-warstory.png — "No results found": per-user averaging washed out the burst]

**War-story #2 — the `avg_diversity=1` bug.** The diversity calculation used `mvcount(split(sub,""))` to count unique characters. But `split(sub,"")` returns **every** character including duplicates, so `mvcount` equaled the string length, and `unique_chars / length` = **1.0 for every single row** — structurally, not by coincidence.

The fix: `mvdedup` to collapse to distinct characters before counting.

```spl
index=main source="dns_logs.csv"
| eval sub=mvindex(split(query,"."),0)
| eval sub_len=len(sub)
| eval chars=split(sub,"")
| eval unique_chars=mvcount(mvdedup(chars))
| eval char_diversity=round(unique_chars/sub_len,2)
| eval suspicious=if(sub_len>20 AND char_diversity>0.45,1,0)
| where suspicious=1
| stats count as suspicious_queries, avg(sub_len) as avg_len, avg(char_diversity) as avg_diversity by user src_ip
| where suspicious_queries>5
| sort -suspicious_queries
```

After the fix: **avg_diversity 0.62**, correctly flagging Andrea's 120 tunneling queries.

📸 [screenshot: project3-04-diversity-bug.png — the bug: avg_diversity = 1 for every row]

📸 [screenshot: project3-05-mvdedup-fix.png — mvdedup fix: avg_diversity now 0.62, Andrea's 120 queries flagged]

📸 [screenshot: project3-06-bubble-clusters.png — bubble chart showing the visible gap between normal (short/low) and tunneling (long/diverse) clusters]

**On synthetic data:** 100% recall, 0% false positives — **by construction**. The synthetic tunneling was built to be long-and-diverse and the normal traffic short-and-simple, so a two-threshold rule trivially separates them. This proves the SPL works; it proves nothing about the real world. That is what the next section is for.

### The real test — and why it matters

The detection was tested against a **real, labeled DNS dataset**: 5,559 domains from a public Zenodo research corpus (benign, DGA, and tunneling classes), loaded as `dns_real_sample.csv`.

📸 [screenshot: project3-07-real-inspect-rows.png — inspecting the real dataset: domain,class schema, 620,704 rows before sampling]

📸 [screenshot: project3-08-real-build-sample.png — building a balanced 5,559-row sample: 3,000 benign + 2,000 tunneling + 1,000 DGA]

📸 [screenshot: project3-09-real-upload.png — dns_real_sample.csv uploaded to Splunk]

📸 [screenshot: project3-10-real-raw-events.png — 5,559 real domains indexed]

📸 [screenshot: project3-11-real-class-counts.png — class counts: 0=benign (3,000), 1=DGA (1,000), 2=tunneling (1,559)]

**The result — the detection largely failed.**

📸 [screenshot: project3-12-real-confusion-matrix.png — the AND detector scored against all three real labels]

| Real class | Total | Flagged | Result |
|---|---|---|---|
| Tunneling | 1,559 | **209** | **13.4% recall** — missed 87% |
| Benign | 3,000 | 67 | **2.2% false-positive rate** |
| DGA | 1,000 | 288 | out of scope (see note) |

On synthetic data the rule caught everything. On real tunneling it caught **13.4%**. That gap is the entire lesson.

> DGA (domain-generation-algorithm) domains are a **third, out-of-scope class** — this detector was built for tunneling, not DGA. Some DGA domains are long and diverse enough to trip the same threshold, which is why 288 are flagged. They are shown for completeness, not as a target of this detection.

**Why it failed — the diagnosis.** The missed tunneling domains weren't uniformly "not tunneling-like." Splitting the real tunneling class (class=2) across the two thresholds:

📸 [screenshot: project3-13-real-missed-tunneling.png — stats on the missed tunneling domains]

📸 [screenshot: project3-14-real-separability-2x2.png — the 2×2 split of real tunneling by length and diversity thresholds]

| | diversity ≤ 0.45 | diversity > 0.45 |
|---|---|---|
| **length ≤ 20** | 4 | **725** |
| **length > 20** | **621** | 209 ✅ (only cell AND catches) |

The `AND` of *length > 20* **and** *diversity > 0.45* only catches the bottom-right cell (209). It **rejected 1,346 real tunneling domains that satisfied only one signal**: 725 were short-but-diverse, 621 were long-but-low-diversity. (Of the 1,350 total missed, 1,346 satisfied exactly one signal; 4 satisfied neither.)

**War-story #3 — the OR "fix" that made it useless.** The obvious next move: relax `AND` to `OR`. That pushed recall to 99.7% — but also pushed the benign false-positive rate to **99.7%**. It flagged essentially everything. Useless.

📸 [screenshot: project3-15-real-OR-fails.png — the OR variant: 99.7% recall but 99.7% false positives]

**The real conclusion.** The malicious class is **bimodal** on these two features — two distinct clusters (short-but-diverse, long-but-low-diversity), each overlapping benign traffic on its own axis. No single axis-aligned AND/OR box separates both clusters from benign. Simple two-threshold boolean logic **cannot** cleanly separate real tunneling; it needs **weighted scoring or ML**. This is documented as future work — **not faked with a tuned number** to make the synthetic result look like it held up.

> **Side note on features.** Some earlier queries in this project referenced a pre-existing `entropy` field (Shannon entropy, ~2–5 bits). The final detection uses **character diversity** (a 0–1 unique-char ratio), computed live. These are two different statistics; the working detection and the diagnosis above use character diversity throughout.

**War-story #4 — the scatter-chart battle.** Six attempts to render a two-color length-vs-diversity scatter in Splunk's classic visualization engine, each failing differently. The classic chart engine maps columns to axes by rigid rules that don't fit "two overlapping clusters in continuous 2D space." The bubble chart (project3-06) with its visible cluster gap does the job the scatter couldn't; the separation argument is carried by the 2×2 table above, not by a scatter plot.

**MITRE ATT&CK.** T1071.004 (Application Layer Protocol: DNS), T1048.003 (Exfiltration Over Unencrypted Non-C2 Protocol), T1568 (Dynamic Resolution).

**Limitations & evasion.**
- Two features are not enough to separate real tunneling from benign traffic — proven above, not asserted.
- An attacker who keeps subdomains short *or* low-diversity (but not both) evades the AND rule entirely — and 1,346 real samples did exactly that.
- Query-volume, timing, and NXDOMAIN-rate features (not used here) would likely help; a weighted or ML model is the real path forward.
- No detection of tunneling over DoH/DoT, which never hits a plaintext DNS log at all.

## Overall Limitations & Honesty Notes

- **All lab data is synthetic** except the abuse.ch IOC feed (Project 2) and the Zenodo DNS corpus (Project 3's real test).
- **Thresholds are hand-tuned, not baselined** against production telemetry.
- **"0 false positives" on synthetic data is by construction**, not a real-world validation.
- **Project 1's non-geo event categories are seeded labels**, not computed detections; only the Haversine geo-velocity logic is a real detection.
- **Project 2 is IOC/IP-match, not behavioral C2 detection.**
- **Project 3's detection largely failed on real data (13.4% recall)** — documented in full, with the correct conclusion that simple boolean thresholds are insufficient and weighted/ML scoring is required.

The failures are the point. A detection that works on data built to make it work proves little; a detection tested against reality, shown to fall short, and correctly diagnosed demonstrates the actual skill of detection engineering.
